# Правила работы агента (AGENT_RULES)

Данный документ представляет собой обязательное операционное соглашение (operational contract) для любого AI-агента, работающего с репозиторием. Правила направлены на предотвращение отклонений от утвержденной архитектурной стратегии.

---

## 1. Контекст проекта и миссия

Мы создаем модульный конвейер обработки данных (**modular AI pipeline (agent-ready)**) для поиска, анализа, скоринга и аутрича fashion-блогеров в интересах бренда **LD Latte**. 

---

## 2. Жесткие ограничения по архитектуре (Hard Architectural Constraints)

Будущие агенты **не должны** отклоняться от выбранного курса:
1. **Фрейминг системы**: Позиционировать проект исключительно как **modular AI pipeline (agent-ready)**. Избегать формулировок «монолитный AI-агент» или автономных неконтролируемых циклов принятия решений.
2. **Локальность и стек**: Вся логика пишется на Python. Запрещено переносить или перепроектировать пайплайн под внешние no-code/low-code платформы (например, n8n) в качестве основного движка оркестрации.
3. **Стратегия скрапинга**: Основным методом должен быть **self-operated скрапинг**.
   * Запрещено переключать проект на сторонние SaaS-решения (например, Apify) в качестве базовой технологии.
   * Строго соблюдать иерархию: Instaloader (первичный) -> Playwright с авторизованной сессией (резервный) -> Apify (только как аварийный бэкап при жестких банах).
4. **Модели эмбеддингов**: Использовать по умолчанию **Qwen3-Embedding-0.6B** (или 4B). Не менять модель эмбеддингов по умолчанию (например, на `multilingual-e5-large` или сторонние проприетарные API) без явного технического обоснования.
5. **Модели реранкинга**: Кросс-энкодер **BAAI/bge-reranker-v2-m3** является обязательным шагом ранжирования.
6. **Роль VLM**: Модели Qwen2.5-VL / Qwen3-VL должны использоваться **исключительно** на этапе **sanity-check финального шорт-листа из 3–5 кандидатов** для оценки визуальной эстетики. Запрещено строить сквозной мультимодальный поиск (multimodal retrieval) по изображениям на больших объемах данных.
7. **Финальная упаковка**: Результат должен поставляться в виде полноценного **интерактивного веб-сайта (Demo UI)** в папке `app/`. Нельзя снижать требования к упаковке до «только README/отчет».
8. **Границы MVP**: Не расширять функционал на Telegram или YouTube Shorts, а также не автоматизировать отправку сообщений (Direct Message/Email) без прямого указания пользователя.

---

## 3. Базовый рабочий процесс (Workflow)

Агент обязан следовать строгому циклу:
1. **Analyze (Анализ)**: Изучить текущую задачу, файлы и логику.
2. **Plan (Планирование)**: Создать четкий пошаговый план изменений. При планировании изменений обязательно указывать, как сохраняется архитектурная концепция.
3. **Wait for Approval (Ожидание аппрува)**: Представить план пользователю и дождаться подтверждения перед редактированием или запуском кода.
4. **Implement (Реализация)**: Написать чистый код с типизацией и обработкой ошибок.
5. **Verify (Верификация)**: Запустить тесты/скрипты для проверки корректности работы.
6. **Update Docs (Обновление документации)**: Зафиксировать статус и изменения в [STATE.md](file:///c:/Users/Admin/Desktop/LD%20LATTE/docs/STATE.md).

---

## 4. Абсолютные правила разработки (Hard Rules)

* **Не трогай стабильные компоненты** без явного разрешения.
* **Максимум 3 файла за один проход реализации** (за исключением специальных сессий обновления документации, одобренных пользователем).
* **Никакого хардкода секретов**. Все ключи и прокси настраиваются через файл `.env` на основе `.env.example`.
* **Если сомневаешься — спроси**. Если требования или контекст размыты, агент обязан задать вопрос пользователю, а не принимать ключевые решения самостоятельно.
* **Сохранение комментариев**: Сохранять все оригинальные комментарии и docstrings в коде.
